# v0.9.12 — `/ai-pre` 工作流再设计 + BUG 反馈簇收尾

> ⚠️ **文件名提示**：harness 指定的占位名是 `humming-roaming-oasis.md`。按 CLAUDE.md §6 规范，落地前应 rename 为 `2026-05-09-v0.9.12-ai-pre-workflow-redesign.md`。

## Context

Admin 在 BugReportDrawer 提了 4 条围绕 `/ai-pre` 的反馈（B-14 ~ B-17）。背后问题是 `/ai-pre` 在 v0.9.7~v0.9.8 拆 6 子组件 + 加 `/ai-pre/jobs` 历史页后，**信息架构没跟上**：项目和 batch 多起来时无法快速定位「哪些项目接了 ML backend / 哪些 batch 可跑 / 跑过的状态」；同时 `/model-market?tab=failed` 与 `/ai-pre/jobs` 双源真相漂移；batch 状态机与 prediction job 也未级联。打包成 v0.9.12 一并落地，避免拆开破窗。

预期成果：① `/ai-pre` 主页改为「项目卡片 → 项目详情面板（多选 batch + 串/并行）」两层 IA；② 删除 `/model-market?tab=failed` tab；③ batch reset_to_draft 级联清 predictions/jobs；④ 已就绪卡片支持多选删除/重激活；⑤ `ml_backends.extra_params.max_concurrency` 接通 asyncio.Semaphore 限速。

## 关键探索发现（影响范围）

- **`FailedPredictionsTab.tsx` 双引用点**：`apps/web/src/pages/ModelMarket/ModelMarketPage.tsx`（tab 项）+ `apps/web/src/pages/AIPreAnnotate/AIPreAnnotatePage.tsx:36,344`（直接内嵌）。**B-14 不能删该组件文件**，只删 ModelMarket 的 tab 项 + tab 容器逻辑（ModelMarket 简化为单页 = RegisteredBackends）。AIPreAnnotatePage 内嵌的 `<FailedPredictionsTab />` 留到 IA 重构时一并清理。
- **`reset_to_draft` 缺级联**（`apps/api/app/services/batch.py:769-809`）：只清 task.status / TaskLock / review fields，未清 `predictions` / `failed_predictions` / `prediction_jobs`。B-15 第一半症状原因。
- **`check_auto_transitions` 已覆盖 `PRE_ANNOTATED → ANNOTATING`**（`apps/api/app/services/batch.py:698`）。B-15 第二半症状（标注员开始标注未转 annotating）需要先**诊断**：确认调用点是否真触发、task.status 是否真改为 `in_progress`。极可能是某条标注流程（如 sam-tool 模式 / acquire_next_task 早期返回）漏调；先加日志再修。
- **`ml_backends.extra_params.max_concurrency` 字段已存在但零消费**（`apps/api/app/db/models/ml_backend.py:24` JSONB）。B-17 并行调度的并发护盾完全缺失。
- **多选基线参考**：`apps/web/src/pages/Projects/sections/BatchesSection.tsx`（460+ 行）已有成熟多选 + 批量操作 UI 模式（`Set<id>` + 表头全选 + 底部浮窗 + bulk mutations 返回 `{succeeded, skipped, failed}`）。B-16 / B-17 多选 UI 直接套这套模式。
- **HistoryTable 是 B-16/B-17 主战场**：`apps/web/src/pages/AIPreAnnotate/components/HistoryTable.tsx`（278 行）当前已支持「按项目折叠分组 + 搜索 + 排序」，但无 checkbox / 无批量操作 / 无项目卡片化入口。
- **API 缺口**：① 无 `DELETE /batches/{id}/predictions` 批量清 prediction 端点；② 无 `/admin/preannotate-summary?group_by=project` 项目维度聚合；③ 无 batch 状态变更 WS 广播（前端要等 30s `useBatches` refetchInterval 才更新）。

## 实施分阶段（建议 6 个 phase，~3 天）

### Phase 1 · 后端：reset_to_draft 级联 + B-15 诊断（B-15）

**目标**：batch 转回 draft 时彻底清空 prediction 痕迹；B-15 第二症状先诊断后修。

**改动**：
- `apps/api/app/services/batch.py:769-809 reset_to_draft`：在 `delete(TaskLock)` 之后加 3 个 delete：
  - `delete(Prediction).where(Prediction.task_id.in_(select(Task.id).where(Task.batch_id == batch_id)))`
  - `delete(FailedPrediction).where(FailedPrediction.task_id.in_(...))`
  - `delete(PredictionJob).where(PredictionJob.batch_id == batch_id)`
  - 三条 delete 行数累加到返回的 `affected` 计数 / audit log
- `apps/api/app/services/batch.py:691-722 check_auto_transitions`：临时加 INFO 日志（batch_id / 入参 status / 查询结果 / 出参 status），实施时跑端到端：admin 触发预标 → 标注员 acquire → 检查日志确认实际未触发的调用点；定位后**补漏调用**或修条件，**不**改 `pre_annotated → annotating` 这条已正确的逻辑。
- 加 audit log entry：`batch_reset_to_draft` extra_data 加 `{predictions_deleted, failed_predictions_deleted, prediction_jobs_deleted}` 三个计数。

**测试**：
- `apps/api/tests/test_batches.py` 加 1 case：reset_to_draft 后 `predictions` / `failed_predictions` / `prediction_jobs` 都为空，且 ON DELETE CASCADE 不会误伤其他 batch。
- 加 1 case：`pre_annotated` batch 内 task 状态改 `in_progress` → `check_auto_transitions(batch_id)` 后 batch 转 `annotating`（验证回归）。

**验收**：admin UI `/projects/:id/settings` 重置已有预标 batch → 回 `/ai-pre` 主页该 batch 卡片消失；`/ai-pre/jobs` 该 batch 行也消失。

### Phase 2 · 后端：max_concurrency 接通 + bulk delete predictions 端点（B-16/B-17 后端）

**目标**：让 `extra_params.max_concurrency` 真正限速；为 B-16 多选删除 + B-17 项目卡片提供端点支撑。

**改动**：
- **bulk delete 端点**：`apps/api/app/api/v1/admin_preannotate.py` 加 `DELETE /admin/preannotate-queue/bulk`（body `{batch_ids: [uuid], mode: "predictions_only" | "reset_to_draft"}`）。
  - `predictions_only`：仅清 prediction / failed_prediction / prediction_jobs，**不**改 batch.status（"删除已就绪卡片但保留批次状态"语义，但批次已不再 PRE_ANNOTATED 因为 prediction 没了 → 状态也要回 ACTIVE，需 service 层补齐）
  - `reset_to_draft`：复用 Phase 1 service.reset_to_draft 逐条调用（已含级联）
  - 返回 `BulkActionResponse {succeeded: [], skipped: [], failed: [{id, reason}]}`（参考 `BatchesSection` 的 `BulkBatchActionResponse`）
  - 鉴权：`require_role([super_admin, project_admin])`，project_admin 仅可处理自己 owned 项目下的 batch
- **max_concurrency 限速**：`apps/api/app/services/ml_client.py` 加 per-backend `asyncio.Semaphore` 缓存（dict[backend_id, Semaphore]），`MLBackendClient.predict()` 调 `/predict` 前 acquire / 调用后 release。
  - 默认 `max_concurrency=4`（向后兼容现有 4 worker concurrency）
  - 信号量按 `ml_backends.id` 分桶，跨 task 共享（同一 backend 全局总并发受限）
  - schema 侧 `apps/api/app/schemas/ml_backend.py` 已有 extra_params 字段，无需 migrate；UI 注册表单暂不暴露（v0.9.12 不做，留 follow-up）；env 直接编辑 ml_backends 行 extra_params JSON。
- **per-project 聚合端点**：`apps/api/app/api/v1/admin_preannotate.py` 加 `GET /admin/preannotate-summary` 返回 `[{project_id, project_name, ml_backend_id, ml_backend_state, has_ready_batches: int, has_active_batches: int, last_job_at, recent_failures: int}]`，仅返回 `ml_backend_id IS NOT NULL` 的项目。供 B-17 项目卡片渲染。

**测试**：
- `apps/api/tests/test_admin_preannotate.py` 加 3 case：bulk delete `predictions_only` 单 batch / 多 batch / 越权（project_admin 跨项目）。
- 加 1 case：`/admin/preannotate-summary` 返回结构 + 无 ml_backend 项目被过滤。
- `apps/api/tests/test_ml_client.py` 加 1 case：mock 5 个并发 predict 调用 + max_concurrency=2 → 实际同时 in-flight 不超过 2（用 `asyncio.Lock` + 计数器验证）。

## 风险与回滚

- **B-15 第二症状定位失败的兜底**：如果加日志后发现 check_auto_transitions 调用点全覆盖，但状态仍未转 → 极可能是用户感知误差（实际转了但前端 30s refetchInterval 才刷）；此时考虑给 batch.status 变更加 WS 广播（`/ws/batches/project/:id`）—— 但这是新基建，建议留 follow-up，本版不做，文档说明「30s 内自动刷新」即可。
- **max_concurrency Semaphore 死锁风险**：per-backend Semaphore 在 backend 不响应时会卡住所有等待者。需 acquire 套 `asyncio.wait_for(timeout=ML_PREDICT_TIMEOUT)`，超时降级为快速失败 + FailedPrediction 记录。
- **bulk delete predictions_only 模式语义争议**：删了 prediction 但 batch 仍 PRE_ANNOTATED 是矛盾态。本 plan 选「同时回 ACTIVE」消除矛盾；如评审觉得应保留 PRE_ANNOTATED 直到所有 prediction 真的清空（部分清不算），可改方案。
- **ProjectCardGrid IA 改动用户接受度**：admin 已熟悉 stepper 流程；建议保留「+ 新建预标」按钮通往原 stepper，避免一上来全砍。如反馈不佳可加一个 toggle「卡片视图 / 列表视图」localStorage 切换。
- **回滚**：Phase 1（DB 级联删）有破坏性，但仅在 admin 显式触发 reset_to_draft 时执行，无误伤；如出问题可发 hotfix 把 3 条 delete 包到 try/except 容错（保留 task 重置 + 不阻断）。其他 phase 纯前端，回滚成本低。
