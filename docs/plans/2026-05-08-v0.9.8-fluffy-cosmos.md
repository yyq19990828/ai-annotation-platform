> Prediction Job 历史 + v0.9.7 端到端跑通后暴露的隐性 bug 收口

## Context

v0.9.7 Virtual Lynx 把 `/ai-pre` 信息架构和 UX 收齐了，但 2026-05-08 用户首次真实跑端到端预标（不再是 fixture）后暴露 6 处隐性问题，零散到下一版会失焦。本版主线把它们捏成一束做掉，复用度都很高（同一个 worker、同一个 ML backend schema、同一个 `/ai-pre` 页面、同一个 WS 通道）：

| 痛点 | 现状 | 本版动作 |
|---|---|---|
| 预标 job 历史只能看「pre_annotated 当前批次」，已重置 / 失败的 job 完全消失 | `/admin/preannotate-queue` 只返回 batch.status=pre_annotated 的快照 | 新建 `prediction_jobs` 表 + 新端点 + `/ai-pre/jobs` 子页 |
| schema adapter 是 v0.9.7 fix 临时塞进去的，缺前后端契约保护 | `to_internal_shape` 在 `services/prediction.py:11-64`；前端 types 全手写 | 引入 openapi-typescript codegen + 黄金样本单测 + dev 文档 |
| 跑完后切项目 progress 全丢 | `usePreannotationProgress(projectId)` 单项目 WS | 新建全局 WS 通道 + Topbar 徽章 + 切项目 toast |
| 用户能填 `localhost` 当 ML backend URL，跑起来直接 connection refused | `MLBackendCreate.url` 仅 schema URL 校验 | 加 host blacklist validator |

总估时 ~5 工作日（含 migration + 测试 + 文档）。

## B1 · prediction_jobs 表 + worker 写入

### Schema (alembic `0052_prediction_jobs.py`)

```sql
CREATE TABLE prediction_jobs (
  id BIGSERIAL PRIMARY KEY,
  project_id BIGINT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  batch_id BIGINT NOT NULL REFERENCES task_batches(id) ON DELETE CASCADE,
  ml_backend_id BIGINT NOT NULL REFERENCES ml_backends(id) ON DELETE RESTRICT,
  prompt TEXT NOT NULL,
  output_mode VARCHAR(30) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'running'
    CHECK (status IN ('running','completed','failed')),
  total_tasks INT NOT NULL DEFAULT 0,
  success_count INT NOT NULL DEFAULT 0,
  failed_count INT NOT NULL DEFAULT 0,
  started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  completed_at TIMESTAMPTZ,
  duration_ms INT,
  total_cost NUMERIC(10, 4),
  error_message TEXT,
  celery_task_id VARCHAR(64),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ix_prediction_jobs_project_status_started
  ON prediction_jobs(project_id, status, started_at DESC);
CREATE INDEX ix_prediction_jobs_status_started
  ON prediction_jobs(status, started_at DESC);  -- 全局 in-progress 查询
```

### Model & Worker 改动

新建 `apps/api/app/db/models/prediction_job.py`（含 `PredictionJobStatus` enum），在 `db/models/__init__.py` 导出。

修改 `apps/api/app/workers/tasks.py`：
- **L27 `_run_batch` 入口**：`INSERT prediction_jobs (status='running', celery_task_id=self.request.id, ...)` 取回 `job_id` 存为局部变量；通过 `self.request.id` 把 job_id 与 celery task id 关联，便于 `on_failure` 回查
- **task 主循环（~L80-130）**：维护本地 counter `success_count` / `failed_count`；累加 `total_cost`（从 `token_meta.total_cost`，已存在变量）
- **L141-145 结束点**：`UPDATE prediction_jobs SET status='completed', completed_at=now(), duration_ms=int((now-started_at)*1000), success_count, failed_count, total_cost`
- **L156-168 `_BatchPredictTask.on_failure`**：通过 `task_id` 回查 job 行 → `UPDATE status='failed', error_message=str(exc)[:2000], completed_at=now(), duration_ms`

### 测试 `apps/api/tests/test_prediction_jobs_worker.py`

5 case：成功跑通写完整字段 / 部分失败 success+failed 计数对 / on_failure 写 error_message / 重置 batch 后旧 job 不变（历史保留）/ duration_ms 非负。

## B2 · GET /admin/preannotate-jobs

### 端点（新文件 `apps/api/app/api/v1/admin_preannotate_jobs.py`）

```
GET /admin/preannotate-jobs
  ?project_id=<int>
  &status=running|completed|failed
  &from=<iso>&to=<iso>
  &search=<prompt 子串>
  &cursor=<base64>&limit=20  (max 100)
→ { items: PredictionJobOut[], next_cursor: str | None }
```

复用 v0.9.7 alias-frequency 的 cursor 风格（`(started_at, id)` 复合 cursor）。Search 在 `prompt` 字段做 `ILIKE '%xxx%'`（已有 GIN index 复用 / 否则走 seq scan，量小阶段不必加 trigram）。

### 与现有 `/admin/preannotate-queue` 区分

| 端点 | 语义 | 来源 |
|---|---|---|
| `/admin/preannotate-queue` | 当前 pre_annotated 批次快照（v0.9.7 既有，**不动**） | `task_batches WHERE status='pre_annotated'` |
| `/admin/preannotate-jobs` | 全量 job 时间线（含已结束 / 重置 / 失败） | `prediction_jobs` 全量 |

注册：`apps/api/app/api/v1/router.py` 加 `include_router(admin_preannotate_jobs.router)`。

### 测试 `apps/api/tests/test_admin_preannotate_jobs.py`（7 case）

基础列举 / project_id 过滤 / status 过滤 / search 模糊匹配 / from-to 范围 / cursor 翻页跨页 / 空结果 200 + items=[]。

## B5 · ml_backends URL host blacklist

### 修改 `apps/api/app/schemas/ml_backend.py:6-21`

```python
@field_validator("url")
@classmethod
def _reject_loopback_host(cls, v: str) -> str:
    parsed = urlparse(v)
    host = (parsed.hostname or "").lower()
    if host in {"localhost", "127.0.0.1", "0.0.0.0", "::1"}:
        raise ValueError(
            "URL 不能用 loopback host（localhost/127.0.0.1）；"
            "容器内访问宿主机请用 docker bridge IP（如 172.17.0.1）或 service DNS。"
            "默认值参考 GET /runtime-hints.ml_backend_default_url"
        )
    return v
```

`MLBackendCreate.url` + `MLBackendUpdate.url`（仅当非 None 时校验）共用同一个 validator。

### 测试

扩 `apps/api/tests/test_ml_backends.py`（或新建 `test_ml_backend_url_validator.py`）：
- POST url=`http://localhost:8001` → 422 + msg 含「loopback」
- POST url=`http://127.0.0.1:8001` → 422
- POST url=`http://172.17.0.1:8001` → 201
- PUT url=`http://localhost:...` → 422
