# v0.10.1 — Capability 协商基础设施（M1）实施计划

> 父 Roadmap：[`ROADMAP/0.10.x.md`](../../ROADMAP/0.10.x.md) §4.v0.10.1
> 日期：2026-05-14
> 估时：~5 工作日

---

## Context

v0.10.0 已让 sam3-backend 容器化并支持挂到 ProjectSettings，但工作台前端**完全没有消费**任何 backend 的 `/setup` 自描述：用户挂上 sam3 后仍能点击 SmartPoint 工具，触发后端 400（sam3 不支持 point）。同时双 backend 在能力上互不为子集（grounded-sam2 有 point，sam3 有 exemplar），未来要并存 N 个后端时必须先把"能力协商"打通。

v0.10.1 目标：
1. 把两个 backend 的 `/setup` 标准化为 JSON Schema 自描述格式（`supported_prompts` / `supported_text_outputs` / `params` 三个字段）。
2. apps/api 新增 `/projects/{id}/ml-backends/{bid}/setup` 代理端点，便于前端按 backendId 拉能力。
3. 前端落地 `useMLCapabilities()` hook 作为 ML 能力的单一事实源，**本期不消费**（M2 才接 ToolDock），但要在 DevTools 可观测。
4. 落地 `MAX_ML_BACKENDS_PER_PROJECT` env（默认 1），让 `POST /projects/{id}/ml-backends` 在超限时返 409 + `{code, message}`；DB schema 一步到位 1:N，运行时锁 1:1。

本期**不做**的事：Prompt-first ToolDock 重构（M2）、ProjectSettings UI 1:N 改造（M3）、prompt-routing（v0.11+）。

## /setup 标准化目标格式（双后端统一）

```jsonc
{
  "name": "sam3-backend",                       // 必填
  "version": "0.10.1",                          // 必填，backend 镜像版本
  "model_version": "sam3.1",                    // 必填，模型 ckpt 版本
  "is_interactive": true,                       // 保留向下兼容
  "supported_prompts": ["bbox", "text", "exemplar"],
  "supported_text_outputs": ["box", "mask", "both"],
  "params": {                                   // JSON Schema (Draft-07 子集)
    "type": "object",
    "properties": {
      "box_threshold": {
        "type": "number", "minimum": 0, "maximum": 1,
        "default": 0.25, "title": "Box 置信度阈值"
      }
    }
  }
}
```

> grounded-sam2 此前的 `box_threshold/text_threshold/sam_variant/dino_variant` 全部按上述形态平移；sam3 的 `model_variant` / `embedding_cache_size` 同理。两个 backend 都新增固定的 `version`（从 `__version__` 或 env `BACKEND_VERSION` 读）。

## 验收（来自 ROADMAP §4.v0.10.1）

1. 两个 backend 的 `/setup` curl 输出符合新 schema：
   ```bash
   curl http://localhost:9090/setup | jq .  # grounded-sam2
   curl http://localhost:9091/setup | jq .  # sam3
   ```

4. 单测全绿：`uv run pytest apps/api/tests/api/v1/test_ml_backends.py -k "limit"`、`pnpm --filter web test useMLCapabilities`。
5. `pnpm docs:gen-env-vars` 后 `docs-site/dev/reference/env-vars.md` 包含 `MAX_ML_BACKENDS_PER_PROJECT`。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| `/setup` 协议破坏式升级导致旧版前端崩溃 | 升级与前端 hook 同 PR；前端 hook 已写兜底；老 UI 路径（SamSubToolbar）不消费 `/setup`，仍按写死 prompts 工作 |
| 30s 代理缓存导致 backend 热修改 params 后前端看不到 | 缓存 key 含 backendId；解绑/重绑时缓存自然失效；M2 加 invalidate 钩子 |
| `ml_backend_limit` 字段加到 `ProjectOut` 导致序列化变更 | 新字段非破坏；前端旧版忽略 |
| env 锁绕过（DB 直插） | 应用层校验只挡 API 入口，符合 roadmap §6 缓解策略 |
