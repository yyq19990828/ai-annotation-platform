# v0.9.20 · Video Tool Semantics

## Summary

v0.9.20 completes the first M5.0 video workbench pass: video bbox and video track are now separate user tools, new video boxes use the same draw-then-pick-class flow as images, selected video objects can be reclassified directly, and tracks can be transactionally converted into independent `video_bbox` annotations.

## Key Changes

- Add video-only `videoTool = "box" | "track"` state. Image `Tool` stays unchanged so `ImageStage` continues to use the existing `TOOL_REGISTRY`.
- Video toolbar now shows `B` single-frame rectangle and `T` track tools. `B` creates `video_bbox`; `T` creates a new `video_track` or appends a keyframe to the selected track.
- Extend pending drawing state so VideoStage can open `ClassPickerPopover` after drawing and commit either `video_bbox` or `video_track` with the selected class.
- Add `POST /tasks/{task_id}/annotations/{annotation_id}/video/convert-to-bboxes` for transactional track conversion:
  - `operation=copy|split`
  - `scope=frame|track`
  - `frame_mode=keyframes|all_frames`
  - server cap: 5000 generated bboxes per request.
- Reuse one backend interpolation helper for Video Tracks JSON export and conversion.
- Add VideoStage controls for current-frame copy/split, keyframe-row copy/split/delete, and whole-track keyframe/all-frame copy/split.
- Video `1-9` now changes the selected video bbox/track class; with no selected object it still changes `activeClass`.

## Assumptions

- Existing `video_track` data is not migrated; `video_bbox` and `video_track` coexist.
- Full-frame conversion is online-only in v0.9.20 and does not enter the offline queue.
- The 5000 bbox cap is intentionally conservative until a bulk annotation API or async job model is added.
