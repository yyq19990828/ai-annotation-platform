# v0.9.25 · 视频后端帧服务 Wave B

> 日期：2026-05-12
> 范围来源：`ROADMAP/2026-05-12-video-backend-frame-service.md` B2 / B3 / B6 / B7。
> 说明：v0.9.24 正在并行开发，本轮使用 v0.9.25。

## 目标

把已落地的帧时间表 B1 扩展为可被前端 R5.3 和后续 AI worker 复用的后端帧服务。

成功标准：

1. 后端能按视频帧范围懒生成 chunk，并通过 task 路由和 videos facade 返回可缓存的 signed URL。
2. 后端能按 `(task_id|dataset_item_id, frame_index)` 抽取 WebP/JPEG 单帧，缓存到 MinIO，并支持批量 prefetch。
3. manifest v2 暴露 chunk、frame timetable、frame service 的入口，同时保留旧 manifest。
4. chunk / frame cache 有基础指标、TTL 配置和 housekeeping 任务。
5. 文档、CHANGELOG、Roadmap、OpenAPI/前端类型同步更新。

## 不做范围

- 不做 B4 segment 协同和 B5 AI tracker 编排。
- 不引入独立 video service；继续使用 FastAPI + Celery media queue。
- 不做 HLS / DASH / 多码率转码。
- 不做 GOP smart-copy 优化；第一版采用确定性 H.264 fragmented MP4 重编码。
- 不改现有 `GET /api/v1/tasks/{task_id}/video/manifest` 响应。

## 后端实现

- 新增 `VideoChunk` / `VideoFrameCache` 表和 `0057_video_frame_service_assets.py` 迁移。
- 新增 `app/services/video_frame_service.py`：
  - 复用 `VideoFrameIndex` 的 `pts_ms` 做 frame range 与抽帧定位。
  - 无 timetable 时按 `fps` 估算 PTS，不阻断旧视频。
  - 统一生成 MinIO key、presigned URL、ETag、Cache-Control、202 pending 响应。
  - `/videos/{dataset_item_id}` facade 必须映射到当前用户可见的 video task。
- 扩展 media worker：
  - `ensure_video_chunks(dataset_item_id, chunk_ids)`。
  - `extract_video_frames(dataset_item_id, frame_requests)`。
  - `cleanup_video_frame_assets()`，由 Celery beat 每天触发。
- 新增 API：
  - `GET /api/v1/tasks/{task_id}/video/manifest-v2`
  - `GET /api/v1/tasks/{task_id}/video/chunks`
  - `GET /api/v1/tasks/{task_id}/video/chunks/{chunk_id}`
  - `GET /api/v1/tasks/{task_id}/video/frames/{frame_index}`
  - `POST /api/v1/tasks/{task_id}/video/frames:prefetch`
  - 同步提供 `/api/v1/videos/{dataset_item_id}/...` facade。
