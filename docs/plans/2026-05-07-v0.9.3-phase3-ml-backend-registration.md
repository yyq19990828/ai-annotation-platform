# v0.9.3 Phase 3 — 前端接通 ML Backend 注册能力

> 计划文件命名遵循项目 CLAUDE.md 约定（`yyyy-mm-dd-vx.y.z` 前缀）。harness 默认建议名 `roadmap-md-0-9-3-phase-happy-meadow.md` 已弃用。

## Context

**问题**：后端 `POST /projects/{project_id}/ml-backends` 端点早已就位（`apps/api/app/api/v1/ml_backends.py:25-44`，权限 `SUPER_ADMIN|PROJECT_ADMIN`），CRUD + health 五件套齐全；但**前端没有任何创建 / 编辑 / 删除 UI**：

- `ProjectSettingsPage` SECTIONS（`apps/web/src/pages/Projects/ProjectSettingsPage.tsx:20-29`）没有「ML 模型」选项卡
- `GeneralSection.tsx:281-305` 只提供下拉**选择**已注册 backend
- `/model-market` 的 `RegisteredBackendsTab` 是超管**只读**总览
- `/admin/ml-integrations` 是只读 overview

唯一注册路径是 curl / Swagger UI / DB seed，PROJECT_ADMIN 完全无入口。同时，`GeneralSection.tsx:301` 文案「暂无可用 backend；先在『ML 模型』选项卡添加。」与 `CreateProjectWizard.tsx:586`「项目设置 → ML 模型」都指向**不存在**的选项卡（commit `e81eb3e` ROADMAP 已标记此 bug）。

**目标**：在 `ProjectSettingsPage` 新增「ML 模型」选项卡，承载该项目作用域下的 backend 注册 / 编辑 / 删除 / 健康检查；让 PROJECT_ADMIN 自服务接入 ML，让 GeneralSection / CreateProjectWizard 的引导文案变成**有效**指向。

**范围**：本期同步在两个入口接通注册 / 编辑 / 删除能力——
- **入口 A · 项目设置**：PROJECT_ADMIN / SUPER_ADMIN 在自己项目内自服务（主路径）
- **入口 B · `/admin/ml-integrations`**：SUPER_ADMIN 跨项目运维总览也能直接编辑（顺手做，share 同一组 hooks + Modal）

后端表 `ml_backends.project_id` 是 NOT NULL FK（`apps/api/app/db/models/ml_backend.py:9-34`），系统**不存在全局 backend 概念**——所有写操作都走已有的 `POST/PUT/DELETE /projects/{pid}/ml-backends[/{id}]`，每个 backend 自带 `project_id`，路由可直接拼。

## 推荐方案

### 1. 新增 `MlBackendsSection`（项目设置选项卡）

**路径**：`apps/web/src/pages/Projects/sections/MlBackendsSection.tsx`（新文件）

仿 `GeneralSection.tsx` 形状，接 `project: ProjectResponse` props。组件内：

- 顶栏：标题「ML 模型」+ 副标题（说明本 tab 只管本项目作用域 backend）+ 右侧「+ 注册 backend」按钮
- 列表区（来自 `useMLBackends(project.id)`）：每行展示 `name / url / state badge / is_interactive 标记 / last_checked_at`，行尾按钮组：
  - 「健康检查」→ 调 `useMLBackendHealth(project.id)`，pending 时按钮 disabled，结果 toast
  - 「编辑」→ 打开 Modal（复用注册表单组件，传入既有数据进入 edit 模式）
  - 「删除」→ 二次确认（轻量 `confirm()` 或 `Modal` 二段式，沿用项目惯例）→ 调 `useDeleteMLBackend`
- 空态：当 `mlBackends.length === 0` 时展示空状态卡片，CTA 同顶栏按钮
- 权限：根据 `usePermissions().role` 判断；`viewer / annotator / reviewer` 进来后所有写操作按钮 disabled 且 tooltip「需要 PROJECT_ADMIN 权限」（与 `MembersSection` 模式一致）

### 2. 新增 `MlBackendFormModal`（注册 / 编辑共用 Modal）

**路径**：`apps/web/src/components/projects/MlBackendFormModal.tsx`（新文件）

复用项目惯例（Modal 组件 + useState + 手写校验，与 `AssignMemberModal.tsx` 同形态）。字段：

| 字段 | 必需 | 形态 | 说明 |
|---|---|---|---|
| `name` | ✓ | 文本框 | 行内 trim 校验，禁止空 |
| `url` | ✓ | 文本框 | 校验 `http(s)://` 前缀（前端轻校验，重校验留给后端） |
| `is_interactive` | — | Switch | 默认 false，hint「交互式 backend 支持点 / 框 prompt（如 SAM）」 |
| `auth_method` | — | Select | 选项 `none / token`；默认 `none`；选 `token` 时下方显示 `auth_token` 输入 |
| `auth_token` | 条件 | 密码框 | 仅 `auth_method=token` 时可见；编辑模式下显示 placeholder「••• 留空保持原值」（避免回显敏感数据） |
| `extra_params` | — | textarea(JSON) | 默认折叠在「高级」`<details>`；空字符串视为 `{}`，非合法 JSON 阻断提交 |

提交逻辑：
- 创建模式：调 `useCreateMLBackend(project.id).mutateAsync(payload)`，成功后 toast「已注册」+ 关闭
- 编辑模式：调 `useUpdateMLBackend(project.id, backendId).mutateAsync(payload)`（**新 hook**，下文 §3）；编辑模式下 `auth_token` 留空时不发字段（避免覆盖为空）

错误处理：捕获 axios 错误，从 `error.response.data.detail` 取后端 message，渲染为 inline alert（仿 `AssignMemberModal` pushToast 模式）。

### 3. 补齐 `useUpdateMLBackend` / `useDeleteMLBackend` hooks

**路径**：`apps/web/src/hooks/useMLBackends.ts`（编辑现有文件）

参照同文件 `useCreateMLBackend` 模式：

```ts
export function useUpdateMLBackend(projectId: string) {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: ({ backendId, payload }: { backendId: string; payload: MLBackendUpdatePayload }) =>
      mlBackendsApi.update(projectId, backendId, payload),
    onSuccess: () => qc.invalidateQueries({ queryKey: ["ml-backends", projectId] }),
  });
}
export function useDeleteMLBackend(projectId: string) {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (backendId: string) => mlBackendsApi.delete(projectId, backendId),
    onSuccess: () => qc.invalidateQueries({ queryKey: ["ml-backends", projectId] }),
  });
}
```

`MLBackendUpdatePayload` 类型加到 `apps/web/src/api/ml-backends.ts`（`Partial<MLBackendCreatePayload>` 即可）。

**路径**：`apps/web/src/pages/Projects/ProjectSettingsPage.tsx`（编辑）

### 5. `/admin/ml-integrations` RegisteredBackendsTab 加编辑能力

**路径**：`apps/web/src/pages/ModelMarket/RegisteredBackendsTab.tsx`（编辑现有文件）

现状是只读 overview（按 project 分组的表格）。改造点：

- **`ProjectGroup` header 右侧**：在「打开项目设置 →」链接旁加「+ 注册 backend」按钮 → 打开 `MlBackendFormModal`（projectId 取自 `group.project_id`）
- **每行 backend 末尾**：在「最近检查」列后追加「操作」列，包含「健康检查 / 编辑 / 删除」三个图标按钮
  - 健康检查 → 调 `useMLBackendHealth(group.project_id)`，结果 toast
  - 编辑 → 打开 `MlBackendFormModal` 编辑模式（传入 backend 既有数据 + projectId）
  - 删除 → `confirm()` 二次确认 → `useDeleteMLBackend(group.project_id)`
- **数据一致性**：`useCreateMLBackend / useUpdateMLBackend / useDeleteMLBackend` 的 `onSuccess` 同时 invalidate `["admin", "ml-integrations", "overview"]` 与 `["ml-backends", projectId]`，让两个入口写后视图同步刷新——这点写在 hooks 内部，避免每个调用方都要记得双 invalidate

**复用关键**：上面 §1-§3 设计的 `MlBackendFormModal` 已经接受 `projectId: string` 作为 prop，本入口直接传 `group.project_id` 即可，零额外组件成本。
