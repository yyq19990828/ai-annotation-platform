# v0.8.2 — 文档深度优化

## Context

v0.8.0/0.8.1 已经把开发文档分组、ADR 0002-0007、协议契约（ml-backend / ws）、SoT 自动生成（hotkeys.generated.md）落齐；治理合规向（系统设置可编辑、注册统计、自助注销、审计分区归档）也在 v0.8.1 收口。

但文档体系仍有四处明显缝隙：
1. **ADR sidebar 漂移**：`docs/adr/` 7 篇 ADR 没接入 `docs-site/.vitepress/config.ts`，读者只能在 GitHub 看；ADR 互引时只能用 GitHub 绝对链接。
2. **`pnpm docs:build` 不在 PR gate**：仅 `docs-site/**` 路径变更时由独立 `docs.yml` 触发；非 docs PR 修改了 `auth.py` 也不会触发，dead-link / hotkeys SoT 漂移要到下次有人改文档才暴露。
3. **how-to 文档与代码绑定漂移**：`add-api-endpoint.md:39-63` 把 `auth.py:238-264` logout 实现整段贴进文档，无任何机制对齐。
4. **echo ML backend 仅 inline 示例**：协议文档 §8 的 60 行样板想跑就要复制粘贴，"先看再写"的心理门槛仍在。

本期目标：把这四个缝隙以**机制化**的方式补上（脚本 / lint / sidebar 同步 / 可执行样板），让后续文档随代码自然漂移即被 CI 拦下，避免再靠人工巡检。

非目标（推迟到 v0.8.3+）：
- 截图自动化（Playwright + IMAGE_CHECKLIST）— 与 E2E spec 写实共建 fixture，1-2 天深活，本期窗口不足
- getting-started 3 张 GIF 录屏 — 手工录屏成本高，等 Playwright 截图自动化方案落地后批量产出
- frontend 单测 ≥ 25% 持续抬升 — 非文档主题
- fabric.js dead dep 清理 — 非文档，留给下次依赖清理 PR

---

## Scope（5 项，按依赖序）

### 1. VitePress `docs:build` 进 CI gate

**目的**：让所有 PR 都跑文档构建，dead-link / hotkeys SoT 漂移即时阻断。

**改动**：
- `.github/workflows/ci.yml`：在 lint job 之后加 `docs-build` job（或并入 lint），不带 `paths` 过滤
  - 步骤：`pnpm install --frozen-lockfile` → `pnpm docs:build`
  - 实测构建 < 5s，全 PR 普遍跑无成本压力
- `.github/workflows/docs.yml`：删除或降级为发布触发（避免重复构建）
- 由用户在 GitHub branch protection 把 `docs-build` 设为 required check

**验收**：本地 `pnpm docs:build` 通过；故意在 .md 写坏链 push PR，CI 红。

---

### 2. snippet 标记 + checker（让文档代码块跟随源文件漂移即报错）

**标记格式**（HTML 注释包裹，行号驱动）：

```markdown
<!-- snippet:apps/api/app/api/v1/auth.py:238-264 -->
\`\`\`python
@router.post("/logout")
async def logout(...):
    ...
\`\`\`
<!-- /snippet -->
```

**为什么不用 anchor**：anchor 方案要求在源文件加 `# region:logout` 注释，污染 SoT；行号方案精确、与 IDE 跳转语义一致、checker 实现简单。

**新文件 / 改动**：
- 新建 `docs-site/scripts/check-doc-snippets.mjs`
  - 扫 `docs-site/**/*.md`，正则匹配 `<!-- snippet:path:start-end -->` ... `<!-- /snippet -->`
  - 读源文件 `path` 的 `start-end` 区间（含两端），与 markdown 代码块体逐行比较
  - 不一致则打印 diff + 退出 1
- `docs-site/package.json`：加 `"check:snippets": "node scripts/check-doc-snippets.mjs"`；prebuild 链追加（`generate-hotkeys` 之后）
- `docs-site/dev/how-to/add-api-endpoint.md`：L39-63 logout 代码块外包裹 snippet 注释，行号绑定 `apps/api/app/api/v1/auth.py:238-264`

**验收**：本地改 `auth.py:238-264` 任一字符不同步文档，`pnpm docs:build` 失败并打印 diff。

### 3. ADR 接入 docs-site sidebar（mirror 方案）

**方案选择**：mirror 同步（仿 `generate-hotkeys.mjs` prebuild 模式），不走外链。

**理由**：外链 GitHub 会跳出站内导航 + 失去 VitePress 全文搜索 + 无版本快照；mirror 后 ADR 互引可用相对路径，且与代码同生命周期。

**新文件 / 改动**：
- 新建 `docs-site/scripts/mirror-adr.mjs`
  - 复制 `docs/adr/*.md` → `docs-site/dev/adr/*.md`
  - 文件头注入 frontmatter 警告 `> 自动镜像自 \`docs/adr/\`，请勿直接编辑此文件。`
  - 把 `docs/adr/README.md` 转为 `docs-site/dev/adr/index.md`
  - 同时输出 `docs-site/dev/adr/sidebar.generated.json`（{title, link} 列表）
- `docs-site/.vitepress/config.ts`：在 dev 组下加 ADR 子组，items 由 `import sidebar from './adr/sidebar.generated.json'` 注入
- `docs-site/package.json`：prebuild 链改为 `mirror-adr && generate-hotkeys && check:snippets`
- `.gitignore`：加 `docs-site/dev/adr/` + `*.generated.json`

**验收**：本地 `pnpm docs:dev` sidebar 出现 ADR 入口（7 + 1 篇）；点击渲染、内部跳转正常；故意把 `docs/adr/` 删一篇，build 后 sidebar 自动减少。

### 5. ADR-0008 草稿：批次 admin-locked 状态机扩展

**目的**：把 ROADMAP A §批次状态机二阶段「`annotating → active` 暂停」的难点 (scheduler 死锁) 文字化，落 ADR 不实现代码。

**新建** `docs/adr/0008-batch-admin-locked-status.md`，`status: proposed`，大纲：

1. **Context**：`BatchService.check_auto_transitions`（`apps/api/app/services/batch.py:661-692`）基于 task 状态自动推进 active↔annotating↔reviewing；运维需暂停批次时无 escape hatch，scheduler 看到 in_progress task 立刻把 batch 拉回 annotating，等于没暂停。
2. **Decision**：引入 batch 级 `admin_locked: bool` 字段，与 7 态 status 枚举正交。`check_auto_transitions` 起始处 `if batch.admin_locked: return` 短路。
3. **State machine 图**（mermaid）：7 态 + admin_locked 正交维度。
4. **API 影响**：新增 `POST /batches/{id}/lock`、`POST /batches/{id}/unlock`，权限 admin only；前端批次列表显示 lock 徽标。
5. **Migration**：alembic 加列 `admin_locked BOOLEAN NOT NULL DEFAULT FALSE`。
6. **Alternatives rejected**：
   - (a) 新增 PAUSED 状态枚举值 — 污染状态机基数，下游统计 / 看板 / 报表都要跟改
   - (b) task 级锁 — 粒度错（运维想暂停的是项目级而非单题）
7. **Consequences**：scheduler 加锁检查；审计日志埋点 lock/unlock；通知 owner 批次被锁。
8. **不在本 ADR 范围**：实现代码（v0.9 跟进，需先评估 scheduler 测试覆盖）。

**验收**：mirror 脚本同步后 sidebar 出现 ADR-0008；`pnpm docs:build` 通过。
